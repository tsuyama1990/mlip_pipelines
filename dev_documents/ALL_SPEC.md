adaptive-mlip 次世代GUIプラットフォーム 要求仕様定義書 (PRD)1. プロジェクト概要と基本理念 (Core Philosophy)本仕様書は、第一原理計算(DFT)、機械学習ポテンシャル(MLIP)、分子動力学(MD)/モンテカルロ(MC)、および遷移状態探索(EON)をシームレスに自動連携させる適応型学習フレームワーク adaptive-mlip のグラフィカル・ユーザー・インターフェース (GUI) の設計・実装要件を定義する。1.1 背景と課題：Foundation Models時代のUXの遅れ現在、計算材料科学およびマテリアルズ・インフォマティクス（MI）の領域は、MACE、CHGNet、M3GNetといった事前学習済みの汎用基盤モデル（Foundation Models）の登場により、劇的なパラダイムシフトを迎えている。しかし、これらの高度なアルゴリズムを実際の産業ワークフローに統合する上で、最大のボトルネックとなっているのがUI/UXの設計である。既存の商用・オープンソースGUI（QuantumATK, MedeA, NanoLabo等）は、依然として「CUIスクリプト言語の薄いラッパー（覆い）」というパラダイムから脱却できていない。一見すると直感的なブロック操作に見えても、能動学習の閾値設定や複雑な空間分割、力場のハイブリッド設定といった「本質的な制御」が必要な場面になると、突然Pythonコードの改変やテキストベースのプロパティ編集を強要する「漏れのある抽象化（Leaky Abstractions）」が顕著になる。この「断絶」は、材料開発の最前線に立つ非専門家のエンジニア（実験研究者やプロセス技術者）にとって、極めて高い認知的負荷（Cognitive Load）を生み出している。ユーザーは物理・化学の課題を解きたいのであって、シミュレーションソフトの構文エラーと戦いたいわけではない。1.2 GUI設計の3大原則：エンジニアを「苦役」から解放する本GUIは、初心者が「素早く、適切に（蒸留して）大規模MLIPを構築し、OTF（On-The-Fly）計算を回し続ける」ことを可能にするため、以下の原則を徹底する。目的主導（Intent-Driven）アプローチの徹底:ユーザーに「learning_rate」や「fix gcmc の引数」を入力させる旧来の方式を廃止する。代わりに、「対象材料（Pt-Ni合金表面）」「知りたい物性（界面でのガス拡散）」「計算予算（クラウドコスト・時間）」というドメイン言語のみを入力させ、システム側のインテリジェントなコンパイラ層が、裏側で最適な計算パイプラインとハイパーパラメータを自動プロビジョニングする。完全な抽象化とスマートなトレードオフ管理:能動学習におけるDFT再計算の閾値などを個別に設定させることは、ユーザーに過度な責任を押し付ける設計である。これを「Accuracy (精度追求)」と「Speed (計算予算優先)」を結ぶ単一のスマートスライダーへと抽象化する。ユーザーはリソースのトレードオフを決定するだけでよく、それに伴う複雑な数理的パラメータ調整はシステムが引き受ける。視覚的・セマンティック（意味論的）な状態管理:LAMMPSの難解な領域分割（region, group）を絶対座標で記述することは、エラーの最大の温床である。3Dビューア上の直感的なマウス操作（ペイントやスライス）で空間を分割・タグ付けし、ASE（Atomic Simulation Environment）を介してバックエンドで自動的に構文矛盾のないコマンド群へと変換する。2. システム・アーキテクチャと画面構成2.1 技術スタックと選定理由フロントエンド: Webテクノロジー（React.js, TypeScript）。高い拡張性と将来的なクラウドSaaS展開を見据える。3D可視化: Three.js または NGLViewベース。数十万原子のトラジェクトリを滑らかにレンダリングし、原子ごとの誤差（不確実性）をリアルタイムに色付け（ヒートマップ）表示する。ノードエディタ: React Flow。複雑なDAG（有向非巡回グラフ）の描画と状態管理に使用。バックエンド: Python (adaptive-mlip コアAPI)。構造管理ハブ: ASE。GUIからのセマンティックな入力（タグ）を保持するデータコンテナとして機能。通信: FastAPI / WebSocket。MD計算等の長時間のジョブ状態をリアルタイムでフロントエンドへプッシュ送信する。2.2 3ペイン・レイアウト設計とユーザー導線ユーザーの視線移動（左から右へ）が自然なワークフローになるよう、画面を3分割する。A. 左ペイン: 3Dインタラクティブ・ビューア (可視化とセマンティック操作)機能: 構造の読み込み・確認、および計算中の原子の動きをリアルタイム表示する。領域定義UX: 「Z軸スライダー」や「矩形選択ツール」を提供。ユーザーは金属スラブの「最下層」を選択し、パレットから「Freeze（固定）」プロパティを塗り絵のように付与する。In-situ Monitoring: OTF実行中、MLIPの予測不確実性が高い原子を赤く表示する。これにより、「なぜ今ここでDFTがトリガーされ、時間がかかっているのか（＝未知の構造に遭遇したから）」をユーザーが直感的に納得できる。B. 下部ペイン: ワークフローエディタ (ノードベースのDAG構成)機能: シミュレーションの手順をブロックの連なりとして構築する。UI表現: [初期構造] → [基盤モデル Fine-tuning] → [Active Learning Loop] といった手順を視覚化する。自動構成: ユーザーが「目的」を入力した時点で、推奨されるノード構成がAIにより自動配置され、ユーザーは微調整のみを行う。C. 右ペイン: コンテキスト認識型インスペクタ & 診断パネル機能: 下部ペインで選択されているノードの詳細設定を行う。段階的開示: 初心者には必須項目（温度、圧力、精度スライダー）のみを表示。熟練者のみトグルで高度なオプションを展開する。プレフライト診断: 設定変更のたびにリアルタイム検証し、警告インジケータ（赤・黄・緑）を表示。3. コアワークフロー詳細仕様3.1 Step 1: 構造入力と空間・アンサンブルのタギングLAMMPSを用いた高度なMDシミュレーション（表面反応や吸着など）において必須となる「空間分割」を、GUI上で完全に抽象化する。課題: 最下層を固定し、中間層を熱浴に接続し、表面を自由に動かすためには、テキストで座標を記述した幾つもの region と group コマンドを書く必要があり、エラーの温床であった。GUI操作: ユーザーは左ペインでスライダーを動かし、「ここから下は固定」と視覚的に領域を定義する。バックエンド・マッピング:指定された領域内の原子群に対し、バックエンドのASE Atoms オブジェクトの tags 属性に整数値をエンコードする。lammps_generator.py がこの情報を元に、LAMMPSの region, group, および fix setforce 0.0 等の複雑なコマンドブロックを自動生成する。3.2 Step 2: MLIPの構築とAuto-HPO (ハイパーパラメータ最適化)MACEやM3GNetといった基盤モデルを、ユーザーの特定の系に合わせてファインチューニングするプロセスを直感的にする。課題: 新たなデータを学習した際に元の知識を失う「破滅的忘却」を防ぐためのパラメータ調整は、非専門家には不可能に近い。GUI操作 (Auto-HPOダッシュボード):learning_rate 等を直接入力させるUIを廃止し、「汎用性を維持する」か「特定系に特化させる」かのポリシーを選択させる。バックエンド処理:バックグラウンドで小規模なベイズ最適化（HPO）が走り、複数のハイパーパラメータセットを並行テストする。画面上には「モデルA（汎用重視）」「モデルB（特定系特化）」といった候補がパレートフロント上にプロットされ、ユーザーはワンクリックで最適なモデルを採択する。3.3 Step 3: 能動学習 (Active Learning / OTF) のスマート制御adaptive-mlip の中核となる、MLIPの推論による高速MDと、不確実性が高まった際のDFT(Oracle)による再学習ループの設定を行う。GUI操作 ("Accuracy vs Speed" スライダー):candidate_threshold 等の専門的な変数を隠蔽し、「1 (Speed優先: 計算は速いが破綻リスクあり)」から「10 (Accuracy優先: 慎重な追加学習)」までのスライダーを提供。バックエンド・マッピング:スライダー値に基づき、外挿グレードの閾値やサンプリング間隔が、あらかじめ定義された非線形関数に従って最適値に自動設定される。3.4 Step 4: 動的LAMMPSタスク（GCMC, 蒸着等）の目的ベース設定ガスの吸着平衡（GCMC）や薄膜生成（Deposition）などの高度なタスクを設定する。課題: fix gcmc 等には8つ以上の難解な引数が並んでおり、手書きは困難である。GUI操作: ユーザーはインスペクタ上で「ガス種」「目標圧力」「温度」のみを入力する。バックエンド変換ロジック:入力されたTとPから、熱力学ソルバーが化学ポテンシャル（$\mu$）を自動計算する。乱数シードや分子テンプレートも自動生成され、最終的なスクリプトへと安全にコンパイルされる。4. 信頼性保証とUX強化機能（フェイルセーフ機構）計算リソースの無駄遣いを防ぎ、初心者の挫折を防ぐための強力なフェイルセーフ機構を実装する。4.1 リアルタイム・プレフライトチェック (Run 0 バリデーション)課題: 数時間をかけて準備したジョブを投入した直後、些細な構文エラーや原子の異常接近（衝突）で即クラッシュする事態を防ぐ。仕様:設定変更のたびに、バックグラウンドで run 0 （初期化テスト実行）を走らせる。初期エネルギーが発散していないか、構文エラーがないかを評価。エラーがあれば該当ノードを赤く点滅させ、衝突している原子を3Dビュー上でハイライトする。4.2 ハイブリッド力場の自動相性診断複数の力場（EAMとLJ等）を混合する設定において、ASEのトポロジー情報と指定力場パラメータを照合する。欠落パラメータがあれば警告を出し、推測値による補完を提案する。4.3 リアルタイムモニタリング・ダッシュボード計算中、以下のグラフをリアルタイムでストリーミング描画する。Loss Curve: MLIP再学習ごとの訓練誤差の推移。エネルギーマップ: MD中のエネルギー推移とDFTトリガーポイントの可視化。反応経路図 (EON連携時): 探索された安定点と鞍点のつながりを動的に更新。5. GUIと adaptive-mlip コアシステムの接続仕様将来的な拡張を容易にするため、APIベースの疎結合なアーキテクチャとする。State File の生成: GUIの設定は単一のYAMLファイルにシリアライズされ、再現性が担保される。Orchestratorへの投入: シリアライズされた設定が Orchestrator に渡され、実計算ループが開始される。双方向ストリーミング: IoManager を拡張し、WebSocketを通じてステータスやメトリクスをフロントエンドへプッシュ送信する。6. プロジェクト推進・開発ロードマップフェーズ 1: React FlowとThree.jsを用いた3ペインUIの静的モックアップ作成。フェーズ 2: GUI設定からLAMMPSスクリプトへの自動コンパイル層の開発。フェーズ 3: "Accuracy vs Speed" スライダーとOTF制御ロジックの統合。フェーズ 4: run 0 バリデーションとリアルタイムダッシュボードの完成。フェーズ 5: 非専門家を対象としたUAT実施とUXの最終チューニング。結語本GUIは、エンジニアの「研究目的」と計算ツールの「難解な仕様」を繋ぐ知的なコンパイラである。複雑さを隠蔽し、意思決定の楽しさと直感的なフィードバックを提供することで、adaptive-mlip は世界中の開発者が日常的に材料設計に活用できるインフラへと進化する。
