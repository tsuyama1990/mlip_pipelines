# CYCLE05 Specification

## 1. Summary

CYCLE05 focuses on the "Hierarchical Finetuning" phase of the NextGen architecture. In Phase 01 of the project, when high-fidelity DFT data was generated for a highly uncertain, defective atomic cluster, that data was simply appended to a massive, ever-growing dataset. The entire ACE potential was then painfully retrained from scratch (batch learning). This naive approach resulted in explosive $O(N^3)$ computational scaling. Worse, it led to severe "catastrophic forgetting", where the machine learning potential forgot how to describe simple, stable bulk structures because the training dataset became overwhelmingly dominated by highly distorted, high-energy defect configurations encountered during the active learning halts.

This cycle completely overhauls the `src/trainers/` module to implement a much smarter, hierarchical approach. First, we introduce `finetune_manager.py`, which takes the newly generated, sparse DFT ground truth data and rapidly fine-tunes the PyTorch-based MACE foundation model. Once MACE is "awakened" to the specific local physics of the failure event (e.g., a specific dislocation core), we use it as an ultra-fast oracle to explosively generate thousands of surrogate training points in the immediate vicinity of that failure. Finally, we drastically update `ace_trainer.py` to perform an "Incremental Update" (Delta Learning) on the ACE potential. Instead of training from scratch, it loads the previous `.yace` polynomial weights, intelligently mixes the new massive surrogate dataset with a fixed-size Replay Buffer sampled via D-Optimality (MaxVol) from historical bulk structures, and performs a rapid, computationally cheap gradient descent step to learn the new physics without forgetting the old.

## 2. System Architecture

The architecture intentionally divides the training layer into two distinct, highly specialized responsibilities: Foundation Model Finetuning and ACE Potential Incremental Updating.

The `FinetuneManager` handles the heavy lifting of adjusting massive PyTorch neural network weights using sparse, high-fidelity ground truth data. It must interface tightly with the `mace_run_train` CLI tools to execute this efficiently on GPUs. The `ACETrainer` handles the rapid translation of the massive surrogate dataset (generated by the awakened MACE model) into the highly performant polynomial basis used by LAMMPS. By keeping these concerns separate, we can independently optimize the batch sizes and learning rates for each architecture.

### File Structure Modification

We introduce a new manager for the PyTorch side and heavily refactor the existing Pacemaker wrapper to support incremental state loading.

```text
.
├── src/
│   ├── trainers/
│   │   ├── __init__.py
│   │   ├── **ace_trainer.py**      (Refactored: Implements Incremental Update & Replay Buffer sampling)
│   │   ├── **finetune_manager.py** (NEW: Dedicated logic for MACE PyTorch Finetuning via CLI wrapping)
```

The Orchestrator will now call these two trainers sequentially during Phase 4, managing the flow of data from DFT -> MACE -> Surrogate Generator -> ACE Trainer.

## 3. Design Architecture

The design centers heavily around smart data management (specifically the implementation of the Replay Buffer) and the intricate subprocess execution configuration (the automated generation of the Pacemaker input YAMLs).

### Domain Concepts and Constraints

1. **Replay Buffer Management:** The system must persistently maintain a massive `training_history.extxyz` file containing every structure ever evaluated by DFT. During an incremental update, the system must use D-Optimality (Active Set selection via `pace_activeset` or MaxVol equivalents) to filter this file down to exactly `replay_buffer_size` structures (as defined in `LoopStrategyConfig`). It then concatenates them with the newly generated surrogate data to form the current iteration's training set. This mathematical optimization ensures that the linearly independent, maximally informative structures are retained to prevent catastrophic forgetting.
2. **Incremental ACE Updating:** The `ace_trainer.py` must dynamically generate a sophisticated `fit.yaml` configuration that explicitly points to the previous active learning iteration's `potential.yace` file using the `initial_potential` flag. Furthermore, it must significantly reduce the `max_num_epochs` parameter compared to a Phase 1 cold start, since the model only needs to learn a small, localized perturbation in the energy landscape rather than discovering the entire landscape from scratch.
3. **MACE Finetuning Constraints:** The `FinetuneManager` must enforce strict constraints during neural network training. It must pass specific CLI flags (e.g., `--freeze_body`) to freeze the early, deep representation layers of the foundation model and only update the final readout layers. This critical constraint prevents the foundation model from losing its "universal knowledge" while it rapidly adapts to the local interface physics.
4. **Data Contract Compliance:** Both trainers must strictly consume and produce data in the format of lists of `ase.Atoms` objects containing fully populated `energy` (float) and `forces` (Nx3 numpy arrays) tags, ensuring seamless interaction with the generators and oracles.

## 4. Implementation Approach

The implementation focuses heavily on string templating for complex configuration files and robust I/O management for massive datasets.

1. **Finetune Manager Creation:** Create `src/trainers/finetune_manager.py`. Implement the `finetune_mace(structures, model_path, output_path)` function. Since MACE training requires invoking a highly complex PyTorch training script (`mace_run_train`), we will use `subprocess.run` to call this CLI tool. We must carefully construct the argument list, passing `--freeze_body`, minimizing the learning rate, and pointing it to a temporary HDF5 or EXTXYZ dataset created specifically for this finetuning run.
2. **Replay Buffer Logic:** Open `src/trainers/ace_trainer.py`. Implement a private method `_manage_replay_buffer(new_surrogate_data, history_file_path, buffer_size)`. Read the entire `history_file_path`. If its length exceeds `buffer_size`, utilize `pace_activeset` to extract the mathematically optimal D-Optimality subset. Append `new_surrogate_data`. Write the combined result to `current_train.extxyz`. Crucially, append the original, un-augmented DFT data back to the persistent `history_file_path` for future iterations.
3. **Incremental Training YAML:** Modify the `_generate_fit_yaml` logic. Add a keyword argument `initial_potential_path: Optional[Path] = None`. If provided, inject the `initial_potential: <absolute_path>` key-value pair into the YAML dictionary before dumping it to disk. Ensure the `max_num_epochs` is dynamically scaled down by a factor of 10 if this is an incremental update.
4. **Trainer Subprocess Execution:** Modify the primary `train()` method. Ensure it executes `pace_train`, pointing precisely to the incrementally generated `fit.yaml` and the carefully constructed `current_train.extxyz`. It must handle subprocess errors gracefully and return the absolute path to the newly minted `potential.yace`.

## 5. Test Strategy

Testing this cycle requires verifying incredibly complex dataset sampling algorithms and meticulous subprocess argument generation without actually running multi-hour Machine Learning training jobs on CI servers.

### Unit Testing Approach
We will thoroughly test the data manipulation and YAML generation logic using isolated Pytest functions within `tests/trainers/`.
- **Replay Buffer D-Optimality Selection Test:** Create a dummy `history.extxyz` file on disk containing 100 simple, uniquely identifiable atomic structures. Call `_manage_replay_buffer` passing `new_data` containing exactly 10 unique structures and a `buffer_size` of 50. Since executing `pace_activeset` is heavy, completely mock the active set selector subprocess call. Assert mathematically that exactly 60 structures are returned in the final training set, and that the persistent `history.extxyz` file on disk now contains exactly 110 structures. Assert the subprocess received the correct arguments.
- **Incremental YAML Generation Test:** Call the YAML generation function passing a specific dummy `previous.yace` path. Assert that the generated YAML file contains the exact `initial_potential` key pointing correctly to the specified absolute path, ensuring Delta Learning will successfully engage.

### Integration Testing Approach
We must deeply mock the heavyweight `subprocess.run` calls to prevent GPU initialization during testing.
- **MACE CLI Mocking:** We will use `unittest.mock.patch` to mock `subprocess.run` entirely within the `FinetuneManager`. We will call `finetune_mace` and deeply inspect the `mock_calls` argument list. We will assert that the generated command-line string list explicitly contains the required safety arguments (e.g., `--freeze_body`, `--max_num_epochs=5`, `--lr=0.001`). We will ensure a dummy model file is copied to the expected output location by the mock function to satisfy the return contract.
- **Side-Effect Management:** All `extxyz` writing, reading, and massive file manipulation will be strictly confined to Pytest `tmp_path` fixtures. This absolutely guarantees no residual gigabyte data files pollute the developer's local environment or the CI runner's limited disk space.
