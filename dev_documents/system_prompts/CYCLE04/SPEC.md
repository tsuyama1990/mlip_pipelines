# CYCLE04: Active Learning (OTF) Smart Control Integration Specification

## 1. Summary

CYCLE04 tackles the critical "Accuracy vs. Speed" slider requirement established in the PRD, connecting the translation layer meticulously built during CYCLE01 directly and securely to the master execution engine (`Orchestrator`). This cycle ensures that the incredibly deep, mathematically complex hyperparameters (such as `candidate_threshold`, active sampling intervals, and Replay Buffer dimensionality) generated by the frontend's singular scalar intent (1-10) are correctly, flawlessly, and safely respected by the highly sensitive active learning loop. The orchestrator must never receive conflicting or physically impossible instructions from the web interface, as this would lead to catastrophic divergence in the multi-million atom molecular dynamics simulations.

Crucially, this cycle focuses on the architectural implementation of the highly complex, asynchronous REST API endpoints required to securely start, elegantly pause, and flawlessly resume the Orchestrator's continuous execution loop as background daemon tasks. This ensures that the robust, SQLite database-backed state machine maintains absolute, bit-perfect integrity even when driven asynchronously by a remote, potentially unstable web client that might disconnect or issue contradictory commands in rapid succession. The orchestrator must become a fully managed, stateful service capable of surviving network partitions and client-side failures without ever corrupting its internal checkpoints.

## 2. System Architecture

This cycle deeply focuses on the `src/api` layer interacting directly, via thread-safe constructs, with the core engine residing in `src/core/orchestrator.py` via a newly introduced asynchronous task manager.

**File Structure (ASCII Tree):**
```text
mlip-pipelines/
├── src/
│   ├── api/
│   │   ├── routes.py                   # Significantly extended with robust /orchestrator endpoints
│   │   └── **task_manager.py**             # New critical async manager for tracking Orchestrator runs and locks
│   ├── core/
│   │   ├── orchestrator.py             # Existing Orchestrator (modified to accept pause signals safely)
│   │   └── state_checkpoint.py         # Existing state database (ensured thread-safe for API status queries)
└── tests/
    └── api/
        └── test_task_manager.py        # Comprehensive unit tests for async Orchestrator control, locking, and signal handling
```

The system architecture flow is as follows: When the user definitively clicks "Start Simulation" in the React GUI, the frontend immediately submits a `POST /orchestrator/start` request containing the full, deeply nested `ProjectConfig` (including the flawlessly translated `LoopStrategyConfig`). The FastAPI endpoint instantly instantiates a fresh `Orchestrator` object and safely hands it over to the `task_manager.py`, which begins executing `orchestrator.run_cycle()` in a strictly managed, separate background thread or isolated asynchronous task. Crucially, the user can subsequently issue an asynchronous `POST /orchestrator/pause` or `/orchestrator/resume` request. These requests do not forcefully terminate the thread; instead, they securely set boolean flags that the Orchestrator gracefully and periodically checks to safely halt its own execution at designated physics boundaries without ever corrupting its local SQLite state.

## 3. Design Architecture

**Domain Concepts & Pydantic Models:**
-   **`OrchestratorCommandDTO`**: A strongly typed model residing in `gui_schemas.py` that dictates the exact control signals allowed.
    -   `action`: An Enum strictly limited to (`START`, `PAUSE`, `RESUME`, `ABORT`). This Enum prevents arbitrary command execution vulnerabilities.
-   **`OrchestratorStatusDTO`**: A robust model returned to the API client representing the deeply verified, current state of the backend active learning loop (e.g., `{"status": "RUNNING", "current_iteration": 5, "current_phase": "PHASE3_MD_EXPLORATION", "timestamp": 1690000000.0}`). This serves as the single source of truth for the GUI's status indicators.

**Key Invariants, Constraints, and Validation Rules:**
1.  **Strict Concurrency Control and Locking**: Absolutely only one Orchestrator instance can actively run per `ProjectConfig` root directory at any given moment. This invariant is physically critical to prevent SQLite database locks, file corruption, and race conditions when writing massive binary restart files to the `active_learning` directory. The `task_manager.py` must aggressively enforce a strict, atomic mutex/lock per unique project ID, instantly rejecting any subsequent `START` commands with an HTTP 409.
2.  **Graceful and Deterministic Termination**: The `PAUSE` command must absolutely not kill the running orchestrator process immediately via a blunt force SIGKILL. Doing so would invariably corrupt the database and leave binary trajectory files in an unreadable, shattered state. Instead, it must securely set a thread-safe flag (`self._pause_requested = True`) deep within the Orchestrator's memory. The Orchestrator will autonomously check this critical flag at mathematically safe, predefined boundaries (e.g., strictly between Phase 3 Exploration and Phase 4 Finetuning) and halt cleanly, updating its checkpoint database definitively before gracefully exiting the `while` run loop.
3.  **Idempotency of Commands**: Submitting a `PAUSE` command to an orchestrator that is already safely `PAUSED` must be fully idempotent, resulting in an HTTP 200 OK without triggering any redundant logic or state changes.

## 4. Implementation Approach

**Step 1: Pydantic Schema Definitions and Enum Enforcements**
Define the `OrchestratorCommandDTO` and the comprehensive `OrchestratorStatusDTO` schemas within the designated `gui_schemas.py` file. Leverage Pydantic's strict type coercion to ensure that the command strings provided by the frontend map flawlessly to the required Python Enum objects, raising immediate `ValidationError`s on typos before the orchestrator is even touched.

**Step 2: Task Manager Implementation and Mutex Control**
Create the entirely new `src/api/task_manager.py` module. Implement a robust singleton or a globally accessible dictionary specifically designed to meticulously track active Orchestrator runs (`active_runs: dict[str, asyncio.Task]`). Implement bulletproof logic to safely spawn the `orchestrator.run_cycle()` routine in a fully isolated separate thread utilizing FastAPI's integrated `BackgroundTasks` feature or a strictly managed `concurrent.futures.ThreadPoolExecutor`. Ensure that the background task's ultimate completion, graceful pause, or unexpected fatal failure updates a global, thread-safe status dictionary so the frontend API can reliably poll `/orchestrator/status` without ever hitting a stale cache or a database lock.

**Step 3: Extending the Orchestrator for Graceful Pausing**
Modify the core execution loop within `src/core/orchestrator.py`. Delicately add a thread-safe `self._pause_requested` boolean flag. At the very start of the `while self.iteration < max_iters:` loop, explicitly check `if self._pause_requested: return "PAUSED"`. Furthermore, ensure these checks are sprinkled throughout the loop at safely resolvable boundaries (such as immediately after the deployment of a new neural network potential, before the next massive MD run begins). Provide a highly secure, public method `orchestrator.request_pause()` that the API can safely call to toggle this flag.

**Step 4: FastAPI Endpoint Integration and Route Protection**
Within `src/api/routes.py`, construct the critical `POST /orchestrator/command` endpoint. This complex endpoint takes the validated `OrchestratorCommandDTO`.
If `START`: Aggressively check the `task_manager` dictionary to absolutely ensure no active run lock currently exists for this specific project directory. If clear, securely spawn the task.
If `PAUSE`: Safely retrieve the active orchestrator instance reference from the `task_manager` dictionary and definitively call `request_pause()`.
Implement the `GET /orchestrator/status` endpoint to securely read the persisted state directly from the SQLite checkpoint database, mapping it flawlessly to the `OrchestratorStatusDTO` and returning it to the client, even if the main orchestrator thread is currently spinning at 100% CPU utilization.

## 5. Test Strategy

Testing this deeply asynchronous cycle focuses intensely on thread safety, graceful and deterministic termination, and absolutely perfect state persistence during concurrent, asynchronous execution triggered by remote clients. We must prove the system cannot be corrupted.

**Unit Testing Approach:**
-   **Target:** The `task_manager.py` concurrency control and atomic mutex locking.
-   **Method:** We will attempt to programmatically submit a `START` command for two distinct orchestrator instances entirely simultaneously (using `asyncio.gather`) for the exact same project root path. We will assert categorically that the locking mechanism catches the race condition and that the second request immediately and reliably raises an HTTP 409 Conflict.
-   **Target:** The delicate `Orchestrator.request_pause()` logic and boundary checks.
-   **Method:** We will deeply mock the inner loops of the Orchestrator to execute infinitely without calling external binaries. We will call `run_cycle()` in an isolated thread. We will programmatically wait exactly 0.1 seconds, then urgently call `request_pause()`. We will assert that `run_cycle()` exits cleanly at the next loop boundary and returns the precise string `"PAUSED"` without ever raising an unhandled exception or corrupting the critical `state_checkpoint` SQLite DB.

**Integration Testing Approach:**
-   **Target:** The complete API-to-Orchestrator, multi-threaded control flow.
-   **Method:** Submit an asynchronous `POST /orchestrator/command` with a `START` action. Immediately and aggressively poll `GET /orchestrator/status` and firmly assert the returned status is exactly `"RUNNING"`. Wait 0.5 seconds to allow the thread to spin up, submit a `PAUSE` command via the API, and aggressively poll the status endpoint until it definitively returns `"PAUSED"`, rigorously verifying that the internal SQLite iteration counter was correctly and permanently persisted to disk before the thread halted.
-   **Side-effect Isolation:** The underlying, massively heavy `DynamicsEngine` and `DFTManager` classes will be entirely and aggressively mocked using `unittest.mock` to return instantly. This completely eliminates expensive HPC cluster calls, allowing the complex Orchestrator loop to spin incredibly rapidly in-memory without ever invoking slow sub-processes or writing massive gigabyte-sized `.extxyz` trajectory files to the local disk during the automated CI pipeline.
