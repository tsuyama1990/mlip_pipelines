# CYCLE01: Frontend Mockups & Core API Scaffolding Specification

## 1. Summary

The primary goal of CYCLE01 is to establish the foundational communication layer between the envisioned web-based Graphical User Interface (GUI) and the robust Python-based backend of the Adaptive-MLIP platform. This cycle will construct the essential FastAPI entry points and implement the new Pydantic Data Transfer Objects (DTOs) required to securely accept, parse, and validate the complex state objects emitted by the frontend. The Adaptive-MLIP framework relies on extremely precise hyperparameter settings to orchestrate the integration of Density Functional Theory, Foundation Models, and active learning dynamics. However, exposing these deeply nested parameters to end-users has proven to be a significant barrier to entry, resulting in frequent misconfigurations and catastrophic simulation failures.

Crucially, this cycle tackles the complex translation of the user's high-level intent into the specific, deeply nested configuration parameters that the existing `ProjectConfig` demands. The hallmark of this translation is the implementation of the "Accuracy vs. Speed" slider and the "Computational Budget" limits. Instead of overwhelming users with parameters like `candidate_threshold`, `sampling_structures_per_system`, or maximum orchestrator iterations, the GUI will simply send a scalar accuracy value between 1 and 10, alongside a categorical budget classification (e.g., LOW, HIGH). The backend must intercept these simplistic variables and automatically calculate the mathematically optimal sub-parameters using rigorously defined functions. This abstraction is not merely cosmetic; it is a critical safety feature that guarantees the physical validity of the underlying simulation while absolutely protecting the user from runaway cloud computing costs.

By the conclusion of this cycle, the system will possess a fully functional, highly secure REST API capable of receiving a serialized UI state, validating its structural integrity against common web-based injection attacks (such as path traversals or malicious string payloads), resolving complex mathematical trade-offs derived directly from user intent, and successfully instantiating a ready-to-execute `ProjectConfig` object securely in server memory. This API layer serves as the absolute boundary between the untrusted web frontend and the highly sensitive, computationally expensive HPC core engine, ensuring that only perfectly formatted, physically sound configuration objects ever reach the internal orchestrator.

## 2. System Architecture

This cycle fundamentally modifies the boundary layer of the Adaptive-MLIP system. We will introduce a new top-level `src/api` module housing the core FastAPI application logic and a entirely new schema definition module `src/domain_models/gui_schemas.py`. Additionally, the existing `src/domain_models/config.py` will be safely extended to accept these translated intents via extremely strict `@model_validator` functions, seamlessly ensuring absolute backward compatibility with existing CLI execution workflows.

**File Structure (ASCII Tree):**
```text
mlip-pipelines/
├── src/
│   ├── **api**/
│   │   ├── **__init__.py**
│   │   ├── **main.py**                     # FastAPI application entry point and security middleware config
│   │   └── **routes.py**                   # REST endpoints for config submission and intent validation
│   ├── core/
│   │   ├── orchestrator.py             # Existing Orchestrator (strictly unmodified in this cycle)
│   ├── domain_models/
│   │   ├── __init__.py
│   │   ├── **config.py**                   # Existing schemas (heavily extended with new translation validators)
│   │   └── **gui_schemas.py**              # New GUI-specific Pydantic DTOs for explicit intent mapping
└── pyproject.toml
```

The system data flow dictates that an HTTP POST request carrying a massive, highly structured JSON payload arrives directly at `src/api/routes.py`. The payload is immediately and aggressively serialized into the lightweight validation models defined in `gui_schemas.py`. Once the initial structure is verified (e.g., confirming string properties do not contain malicious path traversals (`../`) and that integers representing the UI sliders fall strictly within their mathematically defined bounds), the `@model_validator(mode="after")` methods integrated deeply into `config.py` are triggered. These validation methods execute the sophisticated mathematical translation of the 1-10 tradeoff slider, modulated directly by the chosen computational budget, into the actual `LoopStrategyConfig` numerical values inherently required by the engine. If validation fails at any stage—either due to a subtle type mismatch, a flagrant security violation, or a physically impossible derived state—FastAPI automatically intercepts the Pydantic `ValidationError` and gracefully returns a structured HTTP 422 error, completely shielding and protecting the core computational orchestrator from malformed or dangerous inputs.

## 3. Design Architecture

The core of this cycle is the rigorous, uncompromising definition of the Pydantic schemas that perfectly represent the user's explicit intent. Because this system actively drives massive, expensive HPC computations, the data modeling must be absolutely flawless, strictly enforcing data types, ranges, and delicate cross-field mathematical relationships to prevent costly runtime errors.

**Domain Concepts & Pydantic Models:**
-   **`GUIStateConfig`**: Defined entirely within `gui_schemas.py`. This is a purely presentational, completely stateless schema specifically designed to uniquely store the positions, zoom levels, and connections of the React Flow Directed Acyclic Graph (DAG) nodes present in the frontend interface. It contains absolutely zero physics or calculation logic. The backend simply receives it, validates its size footprint to prevent memory bloat, and persistently saves it to the database so the user can subsequently reload their visual workspace exactly precisely as they left it.
-   **`WorkflowIntentConfig`**: Defined meticulously in `gui_schemas.py`. This crucial model captures the core scientific and financial objective of the user. Key, heavily validated properties include:
    -   `target_material`: A String (e.g., "Pt-Ni"). Must be strictly sanitized using internal utilities to prevent injection of malicious shell commands or unauthorized path variables.
    -   `property_of_interest`: A strict Enum explicitly capturing the physical goal (`INTERFACE_ENERGY`, `BULK_MODULUS`, `DIFFUSION`). This ensures the system understands the physical constraints it is optimizing for.
    -   `computational_budget`: A strict Enum explicitly capturing the financial limits (`LOW`, `MEDIUM`, `HIGH`). This acts as the ultimate master safety override on execution time.
    -   `accuracy_speed_tradeoff`: An Integer, strictly validated to be bounded between 1 and 10 using Pydantic's `Field(ge=1, le=10)`. This is the primary internal driver of the backend's automatic parameter provisioning and extrapolation grading.
    -   `enable_auto_hpo`: A definitive Boolean. Flags whether the system should autonomously route into the complex Bayesian optimization paths (to be built and tested in later specific cycles).
-   **`ProjectConfig` Extensions**: The existing massive `ProjectConfig` root model situated centrally in `config.py` will be carefully, safely modified to accept an optional `intent` attribute natively typed as `Optional[WorkflowIntentConfig]`.

**Key Invariants, Constraints, and Validation Rules:**
1.  **Strict Security Boundaries**: Absolutely all incoming strings (like `target_material` or any generic user descriptors) must be aggressively validated utilizing centralized utility functions (e.g., rigorously checking for bash shell injection characters, directory traversals, or excessively long payloads that could easily trigger Memory Denial of Service limits on the host machine).
2.  **Trade-off Translation Mathematics**: The complex translation of the 1-10 accuracy slider must definitively follow a predefined, monotonic mathematical scaling curve. For instance, if the slider is explicitly set to 1 (representing Maximum Speed), the resulting `uncertainty_threshold` in `DistillationConfig` might be aggressively high and loose (e.g., 0.15), and the `replay_buffer_size` might be minimal to actively conserve host RAM memory. If set to 10 (Maximum Accuracy), the mathematical threshold becomes extremely tight and restrictive (e.g., 0.02), and the required buffer size becomes massive to aggressively prevent catastrophic model forgetting. This mathematically strict translation logic is encapsulated exclusively within a highly optimized `@model_validator(mode="after")` decorating the `ProjectConfig`.
3.  **Budget Supremacy Rule**: The `computational_budget` Enum serves as an absolute, uncompromising override on simulation limits. Even if a user maliciously or accidentally requests the highest possible accuracy (`accuracy_speed_tradeoff = 10`), if they also configure a `LOW` computational budget, the validation layer must ruthlessly intervene and silently cap the resulting `max_iterations` to a safe, low number (e.g., 100 iterations maximum) and firmly restrict the global timeout seconds. This ensures that abstract intent settings cannot generate infinite active learning loops that financially bankrupt the project.
4.  **Forward/Backward Architectural Compatibility**: The addition of the new `WorkflowIntentConfig` object must be strictly optional (`default=None`). Existing, mature CLI-based test suites, automated organizational CI pipelines, and expert users providing their own full, explicit YAML configurations must absolutely not be broken by the introduction of this new API translation layer. The validator will strictly only override the deeply nested configuration blocks if the `intent` object is actually provided and actively populated in the incoming HTTP API payload.

## 4. Implementation Approach

The implementation will proceed efficiently in four distinct, logically isolated engineering steps, focusing incredibly heavily on strict typing enforcement, aggressive data validation, and total memory safety. We will leverage modern Python 3.12 syntax features and strictly adhere to the project's demanding Ruff formatting rules and McCabe cyclomatic complexity constraints.

**Step 1: Scaffolding the Schema Extensions (`src/domain_models/gui_schemas.py`)**
We will create the completely new file dedicated entirely and exclusively to handling UI-originated payloads. We will precisely define the `GUIStateConfig` to securely hold arbitrary JSON-like dictionaries specifically for React Flow rendering data, deliberately employing Pydantic's `ConfigDict` to enforce strict maximum size limits to prevent payload bombs. Next, we will carefully define the critical `WorkflowIntentConfig`, expertly building out the required Enums for `ComputationalBudget` and `PropertyOfInterest`, alongside the strict numeric slider bounds using Pydantic's `Field(ge=1, le=10)`. We will absolutely ensure `extra="forbid"` is securely set on the `model_config` for all newly created models. This is an incredibly vital security requirement designed to forcibly prevent external users from sending arbitrary, unmapped payload bloat that could lead to severe Denial of Service (DoS) via Memory Exhaustion or unexpected arbitrary dictionary unpacking errors deep within the Python application logic.

**Step 2: Extending `src/domain_models/config.py` with Translation Math**
We will skillfully modify the existing, complex `ProjectConfig` core model to smoothly incorporate the newly defined `WorkflowIntentConfig` strictly as an optional field. The most mathematically complex and critical engineering task here is flawlessly implementing the vital `@model_validator(mode="after")` function. This function will first verify if `self.intent` successfully exists in the payload. If it definitively does, it will execute the rigorous mathematical mapping logic. For example, the derivation of the baseline accuracy threshold could be perfectly implemented as: `calculated_threshold = 0.15 - (self.intent.accuracy_speed_tradeoff * 0.013)`. This rigorously derived floating-point value is then forcefully, immutably injected directly into `self.distillation_config.uncertainty_threshold`, silently overwriting any previous default values to ensure the backend aligns perfectly with the user's high-level request. Immediately following this, the validator will securely check the `computational_budget` Enum. If it equals `LOW`, the validator will forcefully execute a command akin to `self.loop_strategy.max_iterations = min(self.loop_strategy.max_iterations, 50)`, ensuring absolute financial safety regardless of all other complex interacting settings.

**Step 3: Building the FastAPI Gateway (`src/api/main.py`, `src/api/routes.py`)**
We will securely initialize the FastAPI ASGI application directly in `main.py`, properly configuring standard CORS middleware to permit secure connections from the local React frontend development server during UAT testing. In `routes.py`, we will meticulously define a highly robust `POST /config/submit` API endpoint. The endpoint Python signature will strictly demand a top-level `ProjectConfig` object. The FastAPI framework, integrated flawlessly and seamlessly with Pydantic, will automatically handle the massive JSON-to-Python parsing logic, aggressive type coercion, and deep validation tree execution before our handler code even runs. The endpoint handler itself will therefore be remarkably thin and efficient; it will simply return a standard success HTTP 200 message alongside the massive, serialized, fully resolved internal configuration back to the connecting client, decisively confirming that the mathematical translation was perfectly successful and securely accepted.

**Step 4: Centralizing Validation Utilities for Architectural Cleanliness**
To maintain uncompromising adherence to the strict Ruff `C901` (McCabe cyclomatic complexity strictly < 10) codebase constraints, we will proactively extract any duplicated, complex path traversal algorithms or dense regex-based security checks currently living as scattered class methods in `config.py` into a highly optimized, shared, module-level private function (e.g., `_validate_string_security(val: str)` residing inside a new `src/dynamics/security_utils.py` module). This crucial architectural refactoring fundamentally ensures that the massive new GUI schema additions can immediately reuse the exact same rigorous, battle-tested security logic without introducing massive code duplication or accidental variable shadowing, resulting in a significantly cleaner, faster, and dramatically more maintainable validation tier.

## 5. Test Strategy

Testing this foundational cycle focuses exclusively on absolutely guaranteeing mathematical correctness, verifying deep structural schema validation, and testing API routing behaviors, maintaining absolute, strict isolation from the heavy physical computation engines to absolutely ensure sub-second test execution across all CI pipelines.

**Unit Testing Approach:**
-   **Target:** The intensely complex `@model_validator` mathematical functions securely embedded within `config.py`.
-   **Method:** We will programmatically, efficiently instantiate a massive `ProjectConfig` object, explicitly passing an `intent` dictionary set securely to `accuracy_speed_tradeoff = 1` (Maximum Speed) alongside a `computational_budget = HIGH`. We will then rigorously assert that the resulting deeply nested `DistillationConfig.uncertainty_threshold` exactly, perfectly matches the expected high floating-point value (e.g., 0.15000000). We will repeat this exact parameterized test for the extreme edge cases where `accuracy_speed_tradeoff = 10` (Maximum Accuracy) and verify the threshold securely drops to 0.02.
-   **Target:** `WorkflowIntentConfig` boundary enforcement and Budget Overrides.
-   **Method:** We will maliciously attempt to instantiate the intent configuration model passing an `accuracy_speed_tradeoff` of `0` or an impossibly high `11`. We will assert that a native, interceptable Pydantic `ValidationError` is immediately raised, thoroughly verifying that the input boundary logic is physically and mathematically sound. Separately, we will instantiate a configuration demanding `accuracy_speed_tradeoff = 10` but strictly providing a `computational_budget = LOW`. We will decisively assert that the final, validated configuration successfully caps the `max_iterations` to the low safety limit, mathematically proving the "Budget Supremacy Rule" operates flawlessly.
-   **Isolation Guarantee:** Absolutely all unit tests will execute purely in memory. They require absolute zero file I/O operations, absolutely no temporary directories to be created, and absolute zero mocking of heavy external binaries, as they strictly and exclusively test the Pydantic instantiation rules and internal mathematical translation logic, maintaining millisecond execution speeds.

**Integration Testing Approach:**
-   **Target:** The massive FastAPI `POST /config/submit` endpoint and its ASGI parsing pipeline.
-   **Method:** We will utilize the highly efficient, in-memory `fastapi.testclient.TestClient`. We will algorithmically construct a completely filled, massively nested JSON dictionary accurately representing a realistic, heavy GUI payload originating from the massive React frontend. We will actively issue the synchronous POST request against the test server instance.
-   **Assert:** We will categorically verify that the HTTP status code returned to the client is precisely 200 OK. We will then cleanly parse the massive JSON response body and meticulously verify that the deeply nested physical parameters (which were specifically omitted from the initial tiny request payload but dynamically, mathematically generated by the backend intent translation validator) are perfectly present, correctly typed, and mathematically accurate in the final response.
-   **Target:** Endpoint Error Handling and Malicious Security Rejection.
-   **Method:** We will actively send a JSON payload equipped with a known malicious string deliberately containing active path traversals (e.g., `"target_material": "../../etc/passwd"` or creating deeply nested recursive JSON bombs).
-   **Assert:** We will strictly verify that the FastAPI ASGI application automatically, silently catches the deeply nested Pydantic security validation error and cleanly returns a sanitized HTTP 422 Unprocessable Entity status code. We will aggressively assert that the error payload contains a perfectly clear message detailing the security violation without leaking internal stack traces, thereby definitively guaranteeing that malicious payloads can absolutely never penetrate or reach the highly sensitive orchestration computation layers.
